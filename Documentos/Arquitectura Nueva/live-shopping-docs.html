01
Arquitectura General del Sistema
Vista de alto nivel — todos los servicios y sus relaciones
Descripción: El sistema está dividido en tres capas principales: el cliente (Angular SPA), la capa de servicios (Django REST + Django Channels) y la capa de datos/infra. El video de transmisión nunca pasa por el servidor propio — se embebe desde la plataforma del vendedor, reduciendo costos de infraestructura al mínimo.

Diagrama de Despliegue
Infraestructura física y servicios cloud — stack seleccionado
Stack de despliegue: Railway corre el proceso Django (ASGI via Daphne para HTTP + WS). Supabase provee PostgreSQL gestionado con backups automáticos. Upstash provee Redis serverless (pago por operación). Cloudflare R2 almacena imágenes sin costo de egress.

03
Modelo Entidad-Relación (ERD)
Estructura completa de la base de datos PostgreSQL
Modelo multi-tenant: Cada Tenant es un vendedor. Todo el modelo de datos está aislado por tenant. LiveSession es el núcleo: agrupa productos, reservas y pedidos. El estado del pedido sigue una máquina de estados definida en el Diagrama 07.

Diagramas de Secuencia
Las 3 fases del negocio: Pre-transmisión, En vivo, Post-transmisión
Fase 1 — Pre-transmisión: El vendedor configura la sesión con catálogo, stock y enlace de stream antes de ir en vivo.
Fase 1: Pre-Transmisión Secuencia
sequenceDiagram actor V as Vendedor participant FE as Angular Frontend participant API as Django REST API participant DB as PostgreSQL participant R2 as Cloudflare R2 V->>FE: Crea nueva transmisión FE->>API: POST /api/sessions/ API->>DB: INSERT LiveSession (status=draft) DB-->>API: session_id API-->>FE: {session_id, share_link} V->>FE: Agrega productos al catálogo FE->>API: POST /api/catalog/products/ API->>R2: Upload imagen producto R2-->>API: image_url API->>DB: INSERT Product + SessionProduct DB-->>API: product_id V->>FE: Configura stock y precio promo FE->>API: PATCH /api/sessions/{id}/products/ API->>DB: UPDATE SessionProduct DB-->>API: OK V->>FE: Obtiene enlace de transmisión FE->>API: GET /api/sessions/{id}/share-link/ API-->>FE: {url: "app.dominio.com/live/maria-boutique/abc123"} FE-->>V: Enlace listo para compartir
Fase 2 — En vivo: El comprador se conecta vía WebSocket y reserva en tiempo real. El servidor actualiza el stock y notifica a todos los conectados simultáneamente.
Fase 2: Transmisión En Vivo (WebSocket) Secuencia
sequenceDiagram actor C as Comprador actor V as Vendedor participant FE as Angular Frontend participant WS as Django Channels (WS) participant CACHE as Redis (Channel Layer) participant DB as PostgreSQL V->>FE: Inicia transmisión FE->>WS: WS CONNECT /ws/session/{id}/vendor/ WS->>DB: UPDATE LiveSession status=live WS->>CACHE: Publica evento session.started CACHE-->>FE: {type: "session.started"} C->>FE: Entra al enlace compartido FE->>WS: WS CONNECT /ws/session/{id}/viewer/ WS-->>FE: {type: "session.state", products: [...], stock: {...}} C->>FE: Hace clic en "Reservar" FE->>WS: {action: "reserve", product_id: "xxx", qty: 1} WS->>DB: BEGIN TRANSACTION DB->>DB: SELECT stock FOR UPDATE Note over DB: Stock disponible = 3 DB->>DB: INSERT Reservation DB->>DB: UPDATE stock = 2 DB->>DB: COMMIT WS->>CACHE: Publica stock.updated a toda la sala CACHE-->>FE: {type: "stock.updated", product_id: "xxx", stock: 2} Note over FE: Todos los compradores ven stock actualizado en tiempo real WS-->>FE: {type: "reservation.confirmed", reservation_id: "yyy", expires_in: 600} FE-->>C: "¡Reservado! Tienes 10 min para pagar" V->>FE: Destaca producto en pantalla FE->>WS: {action: "feature_product", product_id: "zzz"} WS->>CACHE: Publica product.featured CACHE-->>FE: {type: "product.featured", product: {...}} Note over FE: Panel del comprador resalta el producto destacado
Fase 3 — Post-transmisión: El comprador completa el checkout, paga con Tigo Money y el pedido se procesa de forma asíncrona via Celery.
Fase 3: Post-Transmisión y Pago Secuencia
sequenceDiagram actor C as Comprador participant FE as Angular Frontend participant API as Django REST API participant CELERY as Celery Worker participant TIGO as Tigo Money API participant DB as PostgreSQL participant WA as WhatsApp API C->>FE: Revisa reservas activas FE->>API: GET /api/orders/my-cart/ API->>DB: SELECT Reservations WHERE customer AND expires > now DB-->>API: [{product, qty, price, expires_in}] API-->>FE: Lista reservas activas C->>FE: Completa datos y confirma pedido FE->>API: POST /api/orders/checkout/ API->>DB: INSERT Order (status=pending_payment) API->>DB: INSERT OrderItems API->>DB: UPDATE Reservations status=converted DB-->>API: {order_id, order_number, total} C->>FE: Selecciona Tigo Money FE->>API: POST /api/payments/initiate/ API->>CELERY: queue: initiate_tigo_payment(order_id) CELERY->>TIGO: POST /api/payment/request TIGO-->>CELERY: {payment_id, ussd_code, expires} CELERY->>DB: INSERT Payment (status=pending) CELERY-->>API: {ussd_code: "*123*...#"} API-->>FE: {payment_code, instructions} FE-->>C: Muestra código USSD para pagar TIGO->>API: Webhook POST /api/payments/webhook/tigo/ API->>CELERY: queue: process_payment_confirmation(payment_id) CELERY->>DB: UPDATE Payment status=completed CELERY->>DB: UPDATE Order status=paid CELERY->>WA: Enviar confirmación WhatsApp al comprador CELERY->>WA: Notificar al vendedor nuevo pedido pagado WA-->>C: "✅ Tu pedido #BOL-2025-001 está confirmado"
