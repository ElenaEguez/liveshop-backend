MVP
Definición del MVP
Qué es el mínimo real para tener un cliente pagando y un live funcionando
¿Qué es el MVP de LiveShop?

Un vendedor puede crear su cuenta, cargar productos con stock, abrir una sesión de live y recibir reservas automáticas en tiempo real mientras transmite en TikTok o Facebook. Al terminar el live, tiene un panel con todos los pedidos organizados, puede confirmar stock y enviar un mensaje de WhatsApp al comprador. No necesita Tigo Money integrado. No necesita IA. No necesita analytics avanzados.

El MVP resuelve UN problema concreto: el vendedor deja de perder pedidos por caos post-live.

Regla de oro del MVP: Si un feature no es necesario para que el primer cliente haga su primer live y cierre su primer pedido en la plataforma, no entra al MVP.
● MUST HAVE — Sin esto no hay MVP
▸Registro y autenticación de vendedor (multi-tenant)
▸Catálogo de productos con stock y variantes (talla/color)
▸Crear sesión de live + agregar productos al live
▸Reservas en tiempo real vía WebSocket durante el live
▸Descuento automático de stock al reservar (sin sobreventas)
▸Panel post-live con lista de pedidos organizada
▸Formulario de datos del comprador (nombre, teléfono, dirección)
▸Notificación WhatsApp manual (link directo wa.me)
▸Deploy funcional en Railway (ASGI + Daphne)
● SHOULD HAVE — Fase 2
▸Notificaciones WhatsApp automáticas (Twilio API)
▸Embed de TikTok/FB en el panel del vendedor
▸Estado de pedido (pendiente / confirmado / enviado)
▸Expiración automática de reservas (configurable)
▸Exportar pedidos a Excel
▸Historial de lives anteriores
▸Vista del comprador (página pública del live)
▸Panel de analytics básico (ventas por live)
● COULD HAVE — Fase 3–4
▸Tigo Money integrado (webhook + confirmación automática)
▸Banco Unión QR generado automáticamente
▸IA para parsear "apúntame 2 del azul" desde chat
▸Reportes mensuales automáticos PDF
▸Dominio personalizado por vendedor
▸Multi-usuario por cuenta (hasta 3)
▸Programa de referidos
● WON'T HAVE — Fuera del scope año 1
▸App móvil nativa (iOS/Android)
▸Procesamiento propio de video/stream
▸Marketplace propio (el video siempre se embebe)
▸Integración con pasarelas internacionales (Stripe, PayPal)
▸ERP o sistema de contabilidad propio
▸Logística / delivery integrado
▸Soporte multiidioma

FASE 2
Live Engine + Pagos
Semanas 9–16 · WhatsApp automático, flujo de comprobante, confirmación de pago, analytics
Objetivo: la plataforma se vuelve profesional y autónoma. El vendedor asigna su QR/datos bancarios una sola vez — el sistema notifica al comprador, recibe el comprobante y el vendedor confirma con un click. Sin API de pagos, sin responsabilidad financiera, funciona con cualquier método boliviano. Meta: 3 primeros clientes pagando Bs. 300/mes.

FASE 3
Growth & Escala
Semanas 17–24 · IA boliviana, onboarding guiado, referidos, expansión La Paz
Objetivo: el ciclo completo es automático — reserva → pago → confirmación → actualización de estado → WhatsApp al comprador, sin intervención del vendedor. Meta: 10 clientes pagantes y MRR ≥ Bs. 3,000.

ADR
Decisiones de arquitectura clave
Por qué elegimos X sobre Y — para no revisitar estas decisiones
DECISIÓN TOMADA ALTERNATIVA DESCARTADA RAZÓN
El video no pasa por nuestros servidores — embed de TikTok/FB
Procesar/retransmitir el video nosotros
El streaming de video costaría $200–500 USD/mes desde el arranque. Con embed: $0 de ancho de banda de video.
Django Channels 4 + Upstash Redis para WebSockets
Node.js / Socket.io / Pusher
El equipo tiene Django como stack principal. No tiene sentido agregar un runtime diferente. Upstash Redis serverless escala sin gestión.
PostgreSQL con select_for_update() para control de stock
Redis como contador atómico
El stock tiene que ser consistente con los pedidos y el historial. Redis como contador pierde consistencia transaccional. PostgreSQL garantiza ACID.
Railway para deploy (no VPS propio)
DigitalOcean Droplet, AWS EC2
Railway maneja SSL, CI/CD y reinicio automático. El fundador no pierde tiempo en DevOps. A 100 clientes se evalúa migrar.
Supabase para PostgreSQL (no Railway PG)
Railway PostgreSQL, PlanetScale
Supabase tiene backups automáticos, connection pooling y el free tier más generoso. Reduce riesgo de pérdida de datos.
Pagos sin API — QR del vendedor + comprobante manual
Integración API Tigo Money / Banco Unión
Sin acuerdo comercial, sin webhooks, sin responsabilidad financiera. El vendedor ya tiene su QR. El sistema sólo organiza el flujo. Cero semanas de integración ahorradas.
Plan único Bs. 300/mes (sin PRO/ENTERPRISE)
Multi-plan con features diferenciadas
Un solo plan elimina fricción de elección. Con < 50 clientes, la complejidad de múltiples planes no vale la pena. Revisar en Año 2 con datos reales.
Comisión asesor 25% pago único (sin retención)
25% conversión + 12% retención mensual
Comisión recurrente complica la contabilidad y genera conflictos si el cliente cancela. Pago único es simple, predecible y motiva cierres rápidos.
Variantes como JSONField en Product
Tabla ProductVariant separada
En el MVP, la estructura de variantes varía mucho por producto. JSONField es más flexible para empezar. Migrar a tabla relacional cuando haya datos reales.
