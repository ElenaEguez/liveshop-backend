Problema detectado: error al abrir la app (HTTP 404)
¿Qué veía el usuario? Al abrir la app aparecía el modal "Ha ocurrido un error / Algo salió mal. Por favor, inténtalo de nuevo." de forma persistente, cada vez que se cerraba volvía a aparecer.

¿Qué decían los logs de Android?

Los logs estaban llenos de ruido de GPU que ocultaba el error real. Las líneas importantes eran:

// Error real (oculto entre cientos de líneas de GPU):
grpc error handler: [object Object]
grpcCallWithRetry: attempt 3 failed [object Object]
grpcCallWithRetry: reached max retries (3), throwing [object Object]
Http response at 400 or 500 level, http status code: 404

El [object Object] aparece porque los errores gRPC son objetos y Android WebView no los serializa automáticamente — se necesita JSON.stringify para verlos correctamente en logcat.

Diagnóstico: ¿qué llamada fallaba?

📱 La app abre → el componente de balance se monta en la pantalla principal (Home)
↓
🔗 El hook useWalletBalance llama al servicio gRPC: walletClient.getBobBalance(accountId)
↓
🌐 Se hace un POST al gateway Envoy en:
https://maelstrom.zen-dev.net/wallet.public.WalletServerPublic/getBobBalance
↓
✗ El servidor responde HTTP 404 — la ruta no existe en el proxy Envoy del entorno dev
↓
🔁 El cliente gRPC reintenta automáticamente 3 veces (backoff exponencial: 200ms → 400ms → 800ms)
↓
💥 Los 3 intentos fallan → se dispara el modal global de error en errorHandler.ts
↓
🔄 El usuario cierra el modal → notifyModalClosed() resetea el flag → al recargar el balance, el ciclo se repite
Causa raíz (problema de backend/infraestructura) El método getBobBalance fue agregado recientemente al proto y al cliente frontend, pero el proxy Envoy en el entorno maelstrom.zen-dev.net no tiene configurada esa ruta. El equipo de backend/infra debe agregar el route /wallet.public.WalletServerPublic/getBobBalance en la configuración de Envoy.
¿Qué hace getBobBalance? Es la llamada que obtiene el saldo disponible en BOB (bolivianos) de la cuenta del usuario. Devuelve el balance en micros (ej: 1500000 = 1.50 BOB) que se muestra en la pantalla principal. Hasta que el endpoint sea desplegado en el backend, el balance no podrá cargarse desde ese entorno.

Refactoring aplicado: flujo QR centralizado en contexto
Objetivo del cambio Al tocar el botón Retirar en Home, el usuario ahora pasa exactamente por el mismo flujo de entrada que el botón QR del footer: verificación KYC, modales de estado, apertura del scanner. Antes, Retirar iba directamente a /withdraw sin verificar KYC.

¿Por qué era un problema el diseño anterior?

La lógica de verificación KYC (chequeos de estado, modales) estaba duplicada y distribuida:

  • El botón QR del footer tenía sus propios estados (useState) y modales KYC
  • El botón Retirar de Home tenía un hook separado (useQRPaymentTrigger) con sus propios estados y modales
  • Resultado: dos instancias independientes → potencial de mostrar modales duplicados si se activaban desde distintos puntos

Solución: contexto compartido QrPaymentEntryContext

Se creó un Context Provider que centraliza todo el flujo de entrada QR. Hay una sola instancia de estados y modales para toda la app.

ANTES — Hook por componente
// QRButton usaba:
useQRPaymentTrigger()
// → sus propios useState KYC
// → sus propios modales

// Balance usaba:
useQRPaymentTrigger()
// → sus propios useState KYC
// → sus propios modales

// = 2 instancias separadas
AHORA — Contexto compartido
// Un solo Provider en app-content:
<QrPaymentEntryProvider>
  // → un solo useState KYC
  // → un solo set de modales
</QrPaymentEntryProvider>

// QRButton y Balance consumen:
useQrPaymentEntry()
// = 1 instancia compartida

Flujo completo al tocar Retirar o QR (ahora idéntico)

👆 Usuario toca Retirar (Home) o QR (footer) → llaman startQrPaymentEntryFlow()
↓
🔍 Se consulta el estado KYC del usuario vía getIdentityRegistration(accountId)
↓
🔀 Según el estado devuelto, se muestra el modal correspondiente:
• REJECTED_UNDERAGE → Modal: usuario menor de edad bloqueado
• INACTIVE → Modal: KYC rechazado
• REQUIRES_ACTION → Modal: KYC en revisión
• Otro estado no-ACTIVE → Modal: iniciar verificación MetaMap
• ACTIVE → abre el scanner QR ✓

Pendiente — Acción requerida del equipo de Backend/Infra
El error 404 NO está resuelto — requiere acción en el backend El endpoint getBobBalance no está configurado en el proxy Envoy del entorno dev. Hasta que se corrija, el saldo en la pantalla principal no cargará correctamente en ese entorno.

Acción requerida

Agregar la ruta en la configuración de Envoy en maelstrom.zen-dev.net:

Servicio: wallet.public.WalletServerPublic
Método:   getBobBalance
Ruta:     /wallet.public.WalletServerPublic/getBobBalance

Una vez desplegado en el backend, el balance cargará automáticamente sin ningún cambio adicional en el frontend.
